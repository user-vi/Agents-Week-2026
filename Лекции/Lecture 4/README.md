## Что такое Eval

**Eval (Evaluation)** — это **систематическая проверка качества системы**.

Базовая схема:

> входная задача → работа системы → логика оценки → численный результат

В контексте курса акцент сделан на **автоматизированные eval-процессы**, которые:

- запускаются регулярно,
- встроены в процесс разработки,
- не зависят от ручной проверки «на ноутбуке разработчика»,
- позволяют измерять прогресс и ловить регрессии.

### Три ключевых свойства Eval

| Свойство | Смысл |
|---|---|
| Систематичность | это не разовая ручная проверка, а повторяемый процесс |
| Автоматизация | eval должен запускаться в CI / общей инфраструктуре |
| Измеримость | результат выражается в числах, по которым видны улучшения и деградации |

## Почему agent eval — отдельная дисциплина

Лекция рассматривает эволюцию сложности систем. По слайдам презентации эта эволюция выглядит так:

| Поколение системы | Что делает система | Что оцениваем | Типичный eval |
|---|---|---|---|
| Single-turn LLM | 1 prompt → 1 answer | качество ответа | BLEU, ROUGE, exact match |
| Multi-turn LLM | диалог с памятью, несколько ходов | связность, контекст, следование инструкциям | диалоговые метрики, human eval |
| Agentic systems | цикл reasoning → action → observation, инструменты, изменения среды | полную траекторию, outcome и состояние среды | code-based graders + LLM-as-a-Judge + human review |

По мере усложнения систем eval тоже перестаёт быть «проверкой текста» и становится проверкой поведения.

**AI-агент** в лекции определяется как система на базе LLM, которая **автономно планирует и выполняет задачи с помощью внешних инструментов**.

Главное отличие от обычного LLM-приложения:

- агент работает **в цикле**;
- его нужно оценивать **не только по финальному ответу**, но и по тому,
  - какие действия он совершал,
  - какие инструменты выбрал,
  - какие параметры передал,
  - как обработал результаты,
  - к какому состоянию среды пришёл.

**Ключевой тезис:** для агента нужно оценивать **всю цепочку Reasoning → Action → Outcome**, а не только финальный текст.

## Зачем оценивать агентские системы

### 1. Надёжность и безопасность
Агенты работают в непредсказуемых условиях и могут влиять на внешний мир. Ошибки нужно ловить **до продакшена**.

### 2. Понимание поведения
Важно понимать не только **что** сделал агент, но и **почему** он принял это решение.

### 3. Контроль прогресса и регрессий
При изменении:

- промптов,
- модели,
- инструментов,
- внешних API,

нужна система, которая покажет: **стало лучше, хуже или просто иначе**.

### 4. Эффективность итераций
Хороший eval превращает развитие агента в **управляемый инженерный процесс**.

### 5. Общий язык для обсуждения качества
Eval даёт **объективные численные сигналы**, на которых можно обсуждать компромиссы:

- лучше / хуже,
- дешевле / дороже,
- быстрее / медленнее,
- стабильнее / менее стабильно.

## Почему это сложно

### Основные источники сложности

| Проблема | Почему важна |
|---|---|
| Каскадность ошибок | ошибка на раннем шаге портит все последующие |
| Множественность корректных путей | агент может решить задачу не тем путём, который ожидал автор eval |
| Стохастичность | одинаковый запрос не гарантирует одинаковую траекторию |
| Скрытые ошибки | агент может заявить об успехе, хотя действие не выполнено |
| Побочные эффекты | важно проверять реальное изменение среды, а не только текст ответа |

Отсюда главный вывод секции:

> **Agent eval — это тест всей цепочки**, а не проверка финального сообщения.

## Словарь терминов

| Термин | Значение |
|---|---|
| **Evaluation Suite / Task Suite** | «корзина» задач для системной проверки агента |
| **Task** | один тестовый пример с входами и критериями успеха |
| **Trial** | один полный прогон задачи |
| **Trajectory** | полная запись всех шагов прогона |
| **Outcome** | финальное состояние среды после выполнения, а не только текст ответа |
| **Evaluator / Judge / Grader** | логика, которая присваивает оценку результату |
| **Eval Harness** | инфраструктура запуска, логирования и оценки |

## Общий pipeline оценки

Лекция разбивает eval на **4 логических компонента**:

1. **Корзина задач** — что именно проверяем.
2. **Eval Harness / инфраструктура** — как запускаем агента.
3. **Критерии** — по каким правилам считаем успех.
4. **Грейдинг** — как применяем критерии к конкретным прогонам.

Важно:

- eval-пайплайн — не одноразовая проверка, а цикл улучшений
- слабость любого компонента обесценивает остальные.

Примеры обратной связи:

- найденные ошибки → добавляются как новые задачи
  Если агент ошибся на реальном кейсе или на тесте, этот кейс нужно сохранить в корзину задач.
- нестабильные оценки → требуют пересмотра критериев
  Если сегодня тест говорит «успех», завтра на почти том же прогоне — «неуспех», проблема может быть не только в агенте, но и в самой оценке.
- инфраструктурные артефакты → могут испортить всё измерение.
  Артефакты — это побочные технические эффекты системы eval, которые не связаны с реальным качеством агента, но искажают результат.
  Примеры:
  база данных не сбросилась после прошлого теста;
  один прогон оставил изменённое состояние для следующего;
  таймаут оборвал задачу.
  Тогда вы измеряете не качество агента, а случайные сбои среды.

## Компонент 1. Корзина задач (Evaluation Suite)

**Корзина задач** — это репрезентативный набор сценариев, на которых системно измеряется работа агента.

Именно корзина определяет, **что на самом деле считается качеством**.

Если корзина:

- не отражает реальные пользовательские сценарии,
- слишком проста,
- слишком однородна,

то измеряется не реальное качество, а удобная иллюзия качества.

### Практическая эволюция корзины

Для начала часто достаточно 20–50 задач из реальных проблем — этого хватает, чтобы собрать первый рабочий pipeline.

| Версия | Как выглядит |
|---|---|
| V0 | 5–10 задач из ручного анализа того, что агент вообще должен уметь |
| V1 | запуск агента, поиск ошибок, добавление edge-cases и failure modes |
| V2 | расширение по категориям, балансам и LLM-сгенерированным задачам |
| V3 | фиксированный regression suite, который гоняется на каждый PR / релиз |
| V∞ | постоянное пополнение по багам, логам, support-фидбеку и новым функциям агента |

Полезное правило со слайдов: **всё, что команда проверяет вручную, по возможности надо превращать в формальный тест**.

### Откуда брать задачи

| Источник | Польза |
|---|---|
| Ручной продуктовый анализ | помогает стартовать рано |
| Продакшн-логи | лучше всего отражают реальное распределение запросов |
| Bug tracker / support | дают реальные болезненные кейсы |
| Публичные бенчмарки | дают типовые проверочные сценарии |
| LLM-генерация | помогает быстро масштабировать покрытие |

### LLM-генерация задач
В лекции предлагается генерировать задачи по матрице из трёх осей. Это способ системно придумывать тестовые задачи, а не писать их случайно.

- **фичи** (например, поиск, изменение, отмена, справка) — что именно должен уметь агент. То есть это ответ на вопрос: какую функцию мы тестируем?
- **сценарии** (ошибки, неоднозначность, edge-cases) — в какой ситуации это происходит. Это ответ на вопрос: в каком контексте агент должен справиться?
- **персоны** (новичок, раздражённый пользователь, нечеткий пользователь) — какой пользователь обращается. Люди формулируют запросы по-разному, и агент должен выдерживать это разнообразие. Это ответ на вопрос: как именно пользователь подает задачу?

Это удобно для масштабирования, но:

> каждую сгенерированную задачу желательно дополнительно проверять вручную: решаема ли она, относится ли к домену и соответствует ли целям агента.

### Зачем тегировать задачи
Полезные теги:

- сложность,
- категория,
- навык,
- тип сценария,
- тип пользователя.

Это позволяет:

- собирать сбалансированные regression-наборы,
- анализировать просадки по категориям,
- подстраивать распределение задач под продакшн. (делать набор тестовых задач похожим на то, с чем агент реально сталкивается у пользователей)

### Полная структура задачи
Задача — это не только текст запроса. «задача» = один полностью описанный тестовый сценарий для проверки агента.

| Компонент | Что содержит |
|---|---|
| Запрос пользователя | что хочет пользователь |
| Начальное состояние среды | база, контекст, политики, текущие объекты |
| Ограничения / политики | правила, по которым агент обязан действовать |
| Ожидаемое конечное состояние | что должно реально измениться при успехе |

### Свойства хорошей корзины

| Свойство | Проверка |
|---|---|
| Недвусмысленность | два эксперта должны одинаково решать, прошёл тест или нет |
| Проверяемое состояние среды | успех должен подтверждаться не только текстом |
| Репрезентативный баланс | корзина должна быть не удобной, а реалистичной |
| Reference solution | у задачи должно быть доказуемо существующее решение |
| Двунаправленность | должны быть кейсы и на действие, и на корректный отказ от действия |
| Воспроизводимость | задача должна корректно запускаться после reset в CI |
| Тегированность | нужна для анализа и балансировки |

## Компонент 2. Eval Harness / инфраструктура прогонов

Хороший eval для агента — это не просто "у нас есть список тестов", а полноценная техническая система, которая умеет эти тесты стабильно запускать, честно измерять и сохранять результаты так, чтобы им можно было доверять.
Eval — это **инженерная система**, обеспечивающая:

- воспроизводимость (один и тот же eval должен давать сопоставимый результат при повторном запуске),
- изоляцию (каждый тестовый прогон должен жить отдельно от других и не портить соседние),
- масштабирование (система eval должна уметь запускать много задач, много прогонов и делать это достаточно быстро),
- сбор артефактов (Артефакты — это все полезные следы прогона, которые позволяют потом понять, что произошло).

### Главный принцип

> В eval должен запускаться **тот же агент, что и в продакшене**: тот же код, тот же промпт, те же инструменты.

Меняется не агент, а **окружение**.

### Из чего состоит инфраструктура

| Компонент | Роль |
|---|---|
| Runner | запускает задачи, поддерживает параллелизм, timeout, retry |
| Environment | даёт изолированную тестовую среду |
| Logger / Tracer | сохраняет полную траекторию шага за шагом |
| Iron User | симулирует пользователя в multi-turn сценариях |
| Storage | хранит прогоны, трейсы и метрики |
| Comparator | сравнивает версии прогонов и помогает разбирать различия |

### Варианты среды

| Подход | Плюсы | Минусы | Когда подходит |
|---|---|---|---|
| Полный mock | быстро, дёшево, детерминированно | риск drift-а: mock со временем расходится с реальным API | ранняя разработка, unit-level проверки |
| Полный production | максимально реалистично | дорого, медленно, опасно, зависит от внешних сервисов | финальная проверка перед релизом, точечно |
| Гибрид | лучший баланс контроля и реализма | требует аккуратной инженерной сборки | основной режим для большинства eval-систем |

Презентация отдельно подчёркивает два практических требования:

- даже в mock-режиме данные должны быть **реалистичными**: разные цены, пропуски, edge-cases, sold-out кейсы, неоднозначные ответы.
  (Mock-режим — это когда вместо настоящих внешних систем вы используете их упрощённые замены: фейковую базу, фейковые API, тестовые ответы вместо реальных)
- **политики — часть среды**, их надо версионировать вместе с тестовыми данными.
  (Политики - правила, по которым агент обязан действовать, например "при неоднозначности нужно задать уточняющий вопрос")

### Практический вывод
Обычно нужен **гибридный режим**:

- снапшот или копия данных (берут не реальную базу, а её копию или заранее сохранённое состояние),
- те же кодовые пути (агент в eval должен проходить тот же самый код, что и в продакшне,
  не "специальную упрощенную версию для теста"),
- изолированные инстансы (каждый прогон должен идти в отдельной, чистой среде),
- реалистичная бизнес-логика (правила должны быть такими же, как в реальной системе, а не упрощенными "для удобства"),
- безопасное выполнение без влияния на реальный мир (агент должен делать "почти настоящие" действия, но так, чтобы не затронуть реальных пользователей и реальные системы).

### Минимальный чек-лист инфраструктуры

- запуск на общей инфраструктуре, а не на ноутбуке;
- retry и таймауты - защита от зависаний и повторные попытки после временных ошибок;
- graceful handling ошибок;
- автоматическая выгрузка результатов после каждого прогона;
- параллельный запуск;
- возможность сравнить результаты между версиями;
- чистое состояние среды перед каждым запуском.

## Железный пользователь (Iron User)

### Зачем он нужен
Многие агентские задачи **многоходовые**: агент задаёт уточняющие вопросы, просит подтверждение, получает недостающие данные.

Чтобы тестировать такие сценарии автоматически, нужен **симулятор пользователя**.

### Каким должен быть Iron User

| Требование | Смысл |
|---|---|
| Фиксированная цель | пользователь знает, чего хочет добиться |
| Естественное поведение | отвечает как живой человек, а не как тестовый скрипт |
| Следование сценарию | не уходит в произвольные фантазии |
| Некооперативность в разумных пределах | не должен чрезмерно помогать агенту |

### Основные проблемы Iron User

| Проблема | Последствие |
|---|---|
| Недетерминированность | один и тот же сценарий даёт разные диалоги |
| Галлюцинации | симулятор может выдумывать данные |
| Избыточная кооперативность | агент проходит тест легче, чем в реальности |
| Риск циклов | агент и симулятор могут зациклиться |

### Практические меры защиты

- ограничение на число шагов — заранее задают максимальную длину диалога или траектории, чтобы агент и симулятор не могли уйти в бесконечный цикл из повторяющихся вопросов и ответов.
- дополнительная валидация ответа симулятора — после генерации проверяют, соответствует ли ответ сценарию, не выдумывает ли симулятор факты и не подсказывает ли агенту лишнего.
- повторная генерация невалидного ответа — если ответ симулятора не прошёл проверку, его генерируют заново, чтобы в eval не попадали шум, галлюцинации и случайные поломки сценария.
- сценарные ограничения на поведение пользователя — симулятору явно задают рамки: что он знает, чего не знает, как отвечает, когда должен уточнять и что не имеет права сообщать, чтобы его поведение оставалось правдоподобным и стабильным.
- корректный дизайн тулов: ответы инструментов должны давать агенту достаточно информации, чтобы не провоцировать лишние циклы — инструменты должны возвращать не только факт ошибки или успеха, но и достаточно контекста для следующего шага, иначе агент может снова и снова задавать один и тот же вопрос или повторять один и тот же вызов.

### Стратегии симуляции пользователя из презентации

| Стратегия | Идея | Сильная сторона |
|---|---|---|
| LLM | симулятор сразу отвечает по сценарию | быстро, дёшево |
| ReAct | сначала «думает», потом отвечает | лучше держит сценарий |
| Verify | второй LLM-проход проверяет качество ответа; при провале — регенерация | снижает галлюцинации |
| Reflection | при провале симулятор сначала анализирует ошибку, затем улучшает ответ | самый надёжный вариант, но дороже |

Ключевое правило для промпта Iron User: **не выдумывать информацию, которой нет в сценарии**. Если пользователь по сценарию не знает номер брони, симулятор должен ответить «не помню», а не придумывать значение.

## Компонент 3. Критерии качества

Без чётких критериев любые числа теряют смысл.

### Что делает хороший критерий

Презентация советует **начинать с 2–3 критериев**, а не пытаться сразу измерять всё: команды, которые стартуют с десяти метрик, часто в итоге не получают ни одной надёжной.

| Свойство | Смысл |
|---|---|
| Aligned | проверяет именно цель агента, а не далёкую proxy-метрику |
| Автоматизируемость | его реально проверять в системе |
| Human-readable | другому человеку понятно, что именно оценивается |
| Диагностичность | по провалу понятно, что именно чинить |
| Стабильность | оценка не должна сильно дрейфовать между прогонами |

**Проверочное правило:** если два эксперта не могут стабильно договориться по критерию, значит критерий требует переработки.

### Две группы критериев

| Тип | Что оценивает | Чем обычно проверяется |
|---|---|---|
| Output-based | конечный результат, формат, состояние среды | code-based checks, asserts, state checks |
| Trajectory-based | путь к результату: логику шагов, эффективность, следование политике | чаще LLM-as-a-Judge |

### Примеры критериев для ассистентского агента

| Критерий | Вопрос | Тип |
|---|---|---|
| Полезность | решена ли задача пользователя | чаще output-based |
| Корректность | были ли ошибки, invalid tool calls, исключения | output-based |
| Обоснованность | соответствовали ли действия и параметры диалогу | trajectory-based |
| Эффективность | не было ли лишних шагов / токенов / обращений | trajectory-based |
| Стабильность | насколько повторяем успех на одной и той же задаче | надкритерий, измеряется отдельно |
| Безопасность / этичность | нарушил ли агент политику | может включать отдельный обязательный фильтр |

## Метрики стабильности: Pass@k и Pass^k

В лекции подчёркивается, что агентские системы **недетерминированы**, поэтому один прогон ничего не говорит о стабильности.

### Pass@k
Вероятность того, что **хотя бы один** из `k` прогонов успешен.

Подходит там, где допустимы повторные попытки, например:

- генерация кода,
- search / sampling-ориентированные сценарии.

### Pass^k
Вероятность того, что **все `k` прогонов** успешны. Если вероятность успеха одного прогона равна `p`, то для независимой грубой оценки используется формула `pass^k = p^k`.

Это особенно важно для ассистентских агентов, где от системы ожидается **надёжное, а не случайно удачное** поведение.

### Практический смысл

| Метрика | Что показывает |
|---|---|
| Pass@k | можно ли добиться успеха повторными попытками |
| Pass^k | насколько поведение надёжно и воспроизводимо |

## Компонент 4. Грейдинг

**Грейдинг** — это применение критериев оценки к одному конкретному запуску задачи.
После того как агент выполнил задачу, система проверки смотрит не только на финальный ответ, но и на всю траекторию: какие инструменты были вызваны, с какими параметрами, были ли ошибки, соблюдались ли ограничения, изменилось ли состояние среды так, как ожидалось. Затем по каждому критерию выставляется оценка — например, pass/fail, балл или метка вроде good / bad — и на их основе формируется итоговый результат прогона.

### Три базовых типа грейдеров

| Тип грейдера | Плюсы | Минусы | Когда особенно полезен |
|---|---|---|---|
| Code-based | быстрый, дешёвый, объективный, воспроизводимый, дружит с CI | хрупок к допустимым вариациям, плохо видит смысл и нюансы | формат, state checks, tool-call checks, точные инварианты |
| Human | самый надёжный источник качественной разметки | дорого, медленно, не масштабируется | эталонные оценки, спорные случаи, калибровка judge-систем |
| LLM-as-a-Judge | гибкий, масштабируемый, умеет оценивать open-ended кейсы и нюансы | требует калибровки, стоит денег, недетерминированен | trajectory-based и смысловые критерии |

Практическое правило из презентации: **начинать с code-based проверок, затем добавлять LLM judge там, где код не покрывает смысловые аспекты, и калибровать judge на человеческой разметке**.

### Как проверять качество LLM-судьи
В лекции упоминается метрика **согласованности** между грейдерами, в том числе через **коэффициент Каппы Коэна (Cohen's kappa)**.

Идея простая:

- сравниваем оценки людей между собой;
- затем сравниваем оценки LLM-судьи с человеческой разметкой;
- если согласованность достаточна, судье можно доверять больше.

Зачем нужна именно kappa, а не просто процент совпадений?
Потому что часть совпадений может возникать случайно. Например, если почти всем задачам ставят pass, два оценщика будут часто совпадать даже при плохом качестве оценки. Cohen's kappa пытается вычесть такое случайное совпадение и показать, насколько согласованность действительно содержательная.

#### Числовой пример

Пусть есть **100 прогонов** агента, и два человека независимо ставят каждому прогону одну из двух оценок: `pass` или `fail`.

Получилось так:

- в **70 случаях** оба поставили `pass`,
- в **10 случаях** оба поставили `fail`,
- в **10 случаях** первый поставил `pass`, второй — `fail`,
- в **10 случаях** первый поставил `fail`, второй — `pass`.

Тогда:

- всего совпадений: `70 + 10 = 80`,
- **наблюдаемая согласованность** = `80 / 100 = 0.80`.

На первый взгляд кажется, что это очень хорошо: люди совпали в 80% случаев.

Но посмотрим, как часто они вообще ставят `pass` и `fail`:

- первый оценщик поставил `pass` в `70 + 10 = 80` случаях, а `fail` — в `20`;
- второй оценщик тоже поставил `pass` в `80` случаях, а `fail` — в `20`.

Если бы они ставили оценки **случайно**, но с такими же долями, то ожидали бы совпадения:

- по `pass`: `0.8 × 0.8 = 0.64`,
- по `fail`: `0.2 × 0.2 = 0.04`,
- всего случайно ожидаемое совпадение: `0.64 + 0.04 = 0.68`.

Теперь считаем kappa:

- `P_o = 0.80` — фактическая согласованность,
- `P_e = 0.68` — ожидаемая случайная согласованность,
- `kappa = (P_o - P_e) / (1 - P_e) = (0.80 - 0.68) / (1 - 0.68) = 0.12 / 0.32 = 0.375`.

Хотя простое совпадение было **80%**, значение **kappa = 0.375** говорит, что реальная согласованность не такая уж высокая.
Причина в том, что оба оценщика и так очень часто ставят `pass`, поэтому большая часть совпадений объясняется общей склонностью "ставить pass".

#### Пример, когда kappa лучше

Возьмем другие **100 примеров**:

- в **45 случаях** оба поставили `pass`,
- в **35 случаях** оба поставили `fail`,
- в **10 случаях** первый поставил `pass`, второй — `fail`,
- в **10 случаях** первый поставил `fail`, второй — `pass`.

Тогда:

- совпадений снова `45 + 35 = 80`,
- **наблюдаемая согласованность** снова `80 / 100 = 0.80`.

Но теперь распределения оценок другие:

- первый оценщик поставил `pass` в `45 + 10 = 55` случаях, `fail` — в `45`;
- второй тоже поставил `pass` в `55` случаях, `fail` — в `45`.

Случайно ожидаемое совпадение:

- по `pass`: `0.55 × 0.55 = 0.3025`,
- по `fail`: `0.45 × 0.45 = 0.2025`,
- всего `P_e = 0.505`.

Теперь считаем kappa:

- `P_o = 0.80`
- `P_e = 0.505`
- `kappa = (P_o - P_e) / (1 - P_e) = (0.80 - 0.505) / (1 - 0.505) = 0.295 / 0.495 ≈ 0.596`

Здесь тоже **80% совпадений**, но kappa уже почти **0.6**, то есть согласованность заметно лучше.

Смысл примера в том, что одинаковый процент совпадений сам по себе еще ничего не гарантирует.
Во втором случае оценки распределены более сбалансированно, поэтому случайное совпадение ниже, а kappa — выше.

#### Практическая эвристика из лекции и презентации:

- значение каппы **выше 0.6** считается уже рабочим ориентиром;
- без человеческой разметки нет ground truth, а значит нельзя надёжно оценить качество самого judge.

### Комбинированный грейдинг
На практике чаще используется **гибрид**:

1. Сначала обязательные бинарные проверки - базовые проверки вида пройдено / не пройдено.
   - безопасность - проверяют, не нарушил ли агент критические ограничения,
   - корректность — проверяют, не было ли технических ошибок: не упал ли вызов инструмента, не передал ли агент невалидные параметры, не было ли исключений, таймаутов или очевидно неверных действий;
   - ключевые state checks - изменилось ли состояние среды так, как должно было измениться в успешном сценарии
2. Если они пройдены — считаем взвешенную метрику по остальным критериям.

Это позволяет:

- не пропускать критические нарушения,
- при этом различать «хорошо» и «очень хорошо» по менее жёстким метрикам.

Классические схемы комбинирования из презентации:

| Схема | Как работает | Когда удобна |
|---|---|---|
| Binary (AND) | все грейдеры обязаны пройти | критические требования и safety-checks |
| Weighted | каждый грейдер даёт балл, затем берётся взвешенная сумма | когда нужно сравнивать «насколько лучше» |
| Hybrid | часть грейдеров must-pass, остальные взвешиваются | самый практичный вариант для production |

## Как строить LLM-as-a-Judge

### Итеративный цикл из лекции

1. Ручная экспертная разметка `30–100` примеров.
2. Проверка согласованности между экспертами.
3. Формулировка промпта для судьи.
4. Прогон на новых примерах.
5. Сравнение с человеческой разметкой.
6. Анализ расхождений.
7. Доработка критериев и промпта.
8. Повтор цикла, пока согласованность не станет приемлемой.

### Что должен видеть LLM-судья

Презентация выделяет **4 обязательных компонента judge prompt**:

| Компонент | Зачем нужен |
|---|---|
| Контекст задачи | чтобы понимать цель, входы и ограничения |
| Данные агента | reasoning trace, tool calls, observations, финальный outcome |
| Rubric / инструкция оценки | что именно проверять, по какой шкале и что означает каждый уровень |
| Формат ответа | обычно JSON, чтобы результат можно было автоматически распарсить |

### Почему наивный judge плох
Наивный промпт вида «оцени качество по нескольким критериям сразу» даёт проблемы:

- завышение оценок;
- halo effect: один критерий тянет за собой другой;
- нестабильность между прогонами;
- размытые определения «хорошо / плохо».

### Пять приёмов улучшения judge-системы

| Приём | Зачем нужен |
|---|---|
| Один критерий = один запрос | снижает взаимное влияние критериев |
| Reasoning (объяснить ход своей оценки) перед вердиктом | повышает качество оценки |
| Чёткие рубрики | формализуют, что значит good / bad |
| Escape hatch | позволяет судье честно сказать «недостаточно данных» |
| Few-shot / golden samples | задают эталонные ориентиры через примеры |

## Типы агентских систем и их оценка

Лекция отдельно подчёркивает: **агентские системы бывают разными**, и подходы к eval у них тоже отличаются. Презентация добавляет примеры типичных benchmark-стратегий.

### 1. Conversational agents
Пример из лекции: агент для работы с билетами.

Особенности оценки:

- multi-turn,
- tool use,
- управление состоянием среды,
- важны и качество общения, и факт выполнения задачи.

Типичный combo-grading:

- state-based check: сравнение `expected_goal_state` и фактического состояния;
- LLM rubric judge: тон, соблюдение политики, groundedness;
- transcript constraints: например, ограничение на число ходов.

Характерный бенчмарк: **[τ-bench / τ2-bench](https://taubench.com/)** для airline и retail-сценариев. На таких задачах очень заметно падение метрики `pass^k`: случайный единичный успех не означает надёжного поведения.

### 2. Coding agents
Сильная сторона — высокообъективная проверка.

Обычно можно проверить:

- проходят ли тесты,
- не сломался ли существующий код,
- корректен ли патч.

Базовая схема — **fail-to-pass**: тесты, которые падали до патча, должны пройти после него; при этом ранее проходившие тесты не должны сломаться.

Характерный бенчмарк: **[SWE-bench Verified](https://www.swebench.com/verified.html)** — реальные GitHub issue в Python-репозиториях. Презентация также упоминает **[Terminal-Bench](https://www.tbench.ai/)** для end-to-end задач в Linux terminal.

### 3. Research agents
Работают с веб-поиском, синтезом источников, построением отчётов и многошаговым factual Q&A.

Особенности:

- много открытых и субъективных задач;
- нужно проверять факты, покрытие темы и качество источников;
- обычно нужна human calibration, потому что здесь субъективности больше, чем у других типов агентов.

Типичная схема: точные проверки для фактуальных задач + LLM judge для groundedness, coverage и source quality. В презентации как референсы названы **[BrowseComp](https://openai.com/index/browsecomp/)** и **[GAIA](https://ai.meta.com/research/publications/gaia-a-benchmark-for-general-ai-assistants/)**.

### 4. Computer-use agents
Используют интерфейсы, близкие к человеческим:

- браузер,
- мышь,
- клавиатура,
- ОС и приложения.

Особенности:

- длинные траектории,
- часто визуальный вход,
- высокие требования к sandbox / VM-среде со snapshot / restore,
- проверка через состояние системы: URL, состояние страницы, файловая система VM и т. п.

Типичные бенчмарки: **[WebArena](https://webarena.dev/)** и **[OSWorld](https://os-world.github.io/)**. Презентация подчёркивает, что сильный tool-calling агент не обязан быть сильным GUI-агентом: интерфейс взаимодействия здесь другой.

## Что важно кроме качества

В продакшене важны не только quality-метрики.

| Метрика | Что означает |
|---|---|
| Latency | насколько быстро работает агент |
| Cost | сколько стоят модельные вызовы и токены |
| Error rate | сколько ошибок создаёт инфраструктура, тулзы или сам агент |

Идея лекции:

> зрелая eval-система должна помогать балансировать **качество, стоимость, скорость и стабильность**, а не максимизировать что-то одно любой ценой.

## Культура качества: что делать в production

Финальные слайды презентации добавляют важную практическую мысль: eval — это не только техника, но и **часть инженерной культуры**.

### Что стоит делать с самого начала

| Практика | Зачем нужна |
|---|---|
| Выносить параметры в конфиги | чтобы быстро проводить эксперименты с temperature, prompts, models, tools |
| Ставить quality goals заранее | например: `accuracy ≥ 90%` для категории X при `latency < 5s` |
| Держать 2–5 надёжных метрик | лучше меньше метрик, но с хорошим сигналом |
| Сохранять каждый прогон | версия агента, параметры, дата, per-task метрики должны быть воспроизводимы спустя месяц |
| Разбирать хвосты распределения | худшие и лучшие кейсы обычно полезнее среднего значения |
| Следить не только за средним, но и за разбросом | высокая дисперсия — предупреждающий сигнал даже при хорошем среднем |
| Использовать жёсткие пороги | `accuracy ≥ 85%`, `latency ≤ 3s` — удобные no-go условия для релиза |
| Сначала добиваться потолка качества, потом оптимизировать стоимость | сначала сильная модель и полный контекст, потом сокращение цены |

### Две роли eval

| Роль | Смысл |
|---|---|
| Capability eval | помогает двигать систему вперёд и улучшать способности |
| Regression eval | защищает от деградации при каждом изменении |

### Замыкание цикла через production

Ключевая мысль презентации: **результаты в онлайне и офлайне должны коррелировать**. Поэтому:

- пользовательские логи и feedback должны превращаться в новые eval-задачи;
- баги должны пополнять regression suite;
- eval — это не checkpoint перед релизом, а постоянный цикл улучшений.

## Главные выводы лекции

1. **Eval — это отдельная инженерная дисциплина**, а не вспомогательная ручная проверка.
2. Для агентов нужно оценивать **не только ответ, но и траекторию**.
3. Хороший eval строится из четырёх частей:
   - корзина задач,
   - инфраструктура,
   - критерии,
   - грейдинг.
4. **Корзина задач — центральный артефакт** всей системы оценки.
5. Multi-turn и tool-using агентов нельзя надёжно тестировать без:
   - изолированной среды,
   - трассировки,
   - симулятора пользователя,
   - нескольких прогонов на одну задачу.
6. **Стабильность так же важна, как и разовая успешность**.
7. LLM-as-a-Judge полезен, но требует:
   - рубрик,
   - эталонной разметки,
   - проверки согласованности.
8. В реальной практике eval — это **замкнутый цикл постоянного улучшения**.
