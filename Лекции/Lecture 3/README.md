## 1. Как агенты думают — TAO Loop

### 1.1. Что такое TAO

**TAO (Thought → Action → Observation)** — цикл работы AI-агента. Позволяет агентам динамически корректировать ответы, улучшать понимание задачи и повышать точность результатов.

Схема цикла:

```
Запрос пользователя
       ↓
   THOUGHT — LLM рассуждает и планирует
       ↓
   ACTION — вызов инструмента / генерация ответа
       ↓
   OBSERVATION — результат из окружения
       ↓
   Задача решена? → Нет → возврат к THOUGHT
                  → Да → Финальный ответ
```

### 1.2. THOUGHT — внутреннее рассуждение агента

| Тип | Пример |
|-----|--------|
| Планирование | «Разобью задачу на 3 шага: найти бронирование → проверить политики → обновить» |
| Анализ | «Судя по ошибке API, проблема в формате даты» |
| Саморефлексия | «Мой предыдущий ответ был слишком общим» |

### 1.3. Четыре стратегии рассуждения

#### Chain-of-Thought (CoT)

Пошаговое рассуждение в рамках одной генерации. Без инструментов — только текстовый вывод. Триггерная фраза: «Let's think step by step.»

```
Вопрос: Сколько будет 15% от 200?
Мысль: Давайте пошагово. 10% от 200 = 20, 5% от 200 = 10 → 15% = 30.
Ответ: 30
```

Хорошо подходит для логических и математических задач. **Проблема:** накопление ошибок — если шаг 2 неверен, всё последующее строится на ошибочной основе. Нет внешней обратной связи.

#### ReAct (Reasoning + Acting)

Чередование рассуждений и действий. Агент использует инструменты и может корректировать себя через наблюдения.

```
Thought: Мне нужно узнать погоду в Париже.
Action: search("weather in Paris")
Observation: 18°C, облачно.
Thought: Теперь я знаю погоду, могу ответить.
Action: final_answer("В Париже 18°C и облачно.")
```

**ReAct > CoT** для задач с инструментами, потому что есть возможность коррекции через Observation.

#### Reasoning-модели (o1-style)

Модели вроде OpenAI o1 и DeepSeek R1 — обучены думать перед ответом. Используют специальные токены `<think>...</think>`. В отличие от CoT и ReAct, это **не стратегия промптинга, а свойство модели**, заложенное при обучении.

#### Сравнительная таблица

| Характеристика | CoT | ReAct | Reasoning (o1) |
|---------------|-----|-------|----------------|
| Пошаговая логика | Да | Да | Да |
| Внешние инструменты | Нет | Да | Опционально |
| Коррекция плана | Нет | Да (Observation) | Внутренняя |
| Где задаётся | Промпт | Промпт | Обучение модели |
| Лучше всего для | Логика, математика | Задачи с инструментами | Сложное планирование |
| Стоимость | Низкая | Средняя (N вызовов) | Высокая (reasoning-токены) |

### 1.4. ACTION — форматы вызова инструментов

| Формат | Описание | Пример |
|--------|----------|--------|
| JSON Agent | Действие как JSON-объект | `{"action": "search_flights", ...}` |
| Function-calling | Нативный формат модели | Сообщения с `tool_calls` |
| Code Agent | Генерация исполняемого кода | `results = search_flights(...)` |

Механизм **Stop and Parse**: LLM генерирует JSON действия → `[STOP]` → парсинг action + args → Runtime вызывает инструмент → результат.

### 1.5. OBSERVATION — обратная связь из окружения

Три этапа обработки: **Parse** (определить функцию и аргументы) → **Execute** (выполнить действие) → **Append** (результат → в контекст агента).

| Тип | Пример |
|-----|--------|
| Данные API | `{"flights": [...], "count": 3}` |
| Системная ошибка | `{"error": "rate_limit_exceeded", "retry_after": 30}` |
| Подтверждение | `{"status": "booking_confirmed", "id": "BK-12345"}` |
| Ввод пользователя | «Да, бронируйте этот рейс» |

### 1.6. Структурированный вывод для планирования

Pydantic-схемы позволяют получить от LLM типизированный план:

```python
from pydantic import BaseModel

class SubTask(BaseModel):
    agent_name: str
    description: str
    priority: int

class Plan(BaseModel):
    subtasks: list[SubTask]

structured_llm = llm.with_structured_output(Plan)
plan = structured_llm.invoke("Break the rebooking task into subtasks")
```

### 1.7. TAO Loop в действии: перебронирование рейса

```
1  Thought: Найти бронирование
   Action:  get_booking(user_id="U-1234")
   Obs:     {booking_id: "BK-789", flight: "SU2454"}

2  Thought: Ищу рейсы на завтра
   Action:  search_flights("SVO", "CDG", "04-16")
   Obs:     [{SU2456: 37k₽}, {AF1147: 42k₽}]

3  Thought: Проверю политики
   Action:  get_policy("rebooking")
   Obs:     {fee: 3000₽, same class only}

4  Thought: Предложу варианты пользователю
   Action:  final_answer("SU2456 или AF1147, доплата $40")
```

Итого: 4 итерации TAO, 3 вызова инструментов, 1 финальный ответ.

## 2. Мульти-агентные системы (MAS)

### 2.1. Зачем нужны MAS

Проблема одного агента при росте сложности задач: теряется среди большого количества инструментов, путает контексты подзадач, сложно отлаживать, становится ненадёжным.

**Решение:** специализированные агенты, каждый — эксперт в своей области.

### 2.2. Single-Agent vs Multi-Agent

| | Single-Agent | Multi-Agent |
|---|---|---|
| Сильные стороны | Простота, нет координации, меньше ресурсов | Сложные задачи, специализация, параллелизм, отказоустойчивость |
| Слабые стороны | Ограничен контекстом, путается с множеством инструментов | Overhead координации, сложность отладки |
| Когда выбирать | Простые задачи (2–5 инструментов) | Много инструментов, разнообразная экспертиза |

Ключевые преимущества MAS: более полное покрытие аспектов задачи, взаимная проверка и валидация результатов, маленькие специализированные модели вместо одной большой, устойчивость к сбоям отдельных агентов.

### 2.3. Шесть принципов MAS

1. **Декомпозиция** — разбиение задачи на подзадачи
2. **Специализация** — каждый агент — эксперт в своей области
3. **Коммуникация** — эффективный обмен информацией
4. **Координация** — согласование действий и агрегация результатов
5. **Верификация** — критики проверяют промежуточные результаты
6. **Обратная связь** — коррекция на основе информации от других участников

### 2.4. Три типа взаимодействия между агентами

**Дебаты** — агенты спорят и приходят к консенсусу. Когда нужна точность, фактчекинг, оценка рисков.

```
Agent A: "$450" ↔ аргументы ↔ Agent B: "$382" → Консенсус: "$382.50"
```

**Коллаборация** — агенты работают последовательно (пайплайн). Координатор агрегирует. Когда есть чёткий пайплайн и разнообразные навыки.

```
Agent 1: Flight → Agent 2: Policy → Agent 3: Booking → Координатор агрегирует
```

**Конкуренция** — агенты предлагают разные стратегии, эвалюатор выбирает лучшую. Для креативных задач.

```
Agent A: "дешевле" / Agent B: "быстрее" / Agent C: "комфорт" → Evaluator выбирает
```

### 2.5. Handoff — ключевой паттерн коллаборации

```
User: "Хочу перебронировать и спросить про багаж"
  → Координатор ("2 подзадачи")
    → Rebook Agent (параллельно!) → handoff: {flight: SU2456}
    → Baggage Agent (параллельно!) → handoff: {rules: "23kg"}
  → Координатор → "Рейс SU2456 + $40, багаж 23кг"
```

### 2.6. Архитектуры MAS

**1. Иерархическая (вертикальная)** — координатор наверху, специалисты внизу.

**2. Децентрализованная (горизонтальная)** — все агенты равноправны, общаются напрямую.

**3. Централизованная (звезда)** — центральный роутер направляет запросы к нужному агенту.

**4. Общий пул сообщений** — все агенты читают и пишут в общий буфер.

| Критерий | Иерархическая | Децентрализованная | Централизованная | Общий пул |
|----------|--------------|-------------------|-----------------|-----------|
| Сложность задач | Высокая | Средне-высокая | Низко-средняя | Средняя |
| Отказоустойчивость | Средняя | Высокая ★ | Низкая | Средняя |
| Масштабируемость | Средняя | Высокая ★ | Низкая | Высокая ★ |
| Простота отладки | Высокая ★ | Низкая | Высокая ★ | Средняя |
| Предсказуемость | Высокая ★ | Низкая | Высокая ★ | Средняя |

### 2.7. Протоколы: MCP и A2A

**MCP (Model Context Protocol)** — стандарт связи Agent ↔ Tool. Аналогия: USB-C — универсальный адаптер для подключения AI к инструментам. Выпущен Anthropic, ноябрь 2024.

**A2A (Agent-to-Agent)** — стандарт связи Agent ↔ Agent. Аналогия: Wi-Fi Direct — прямое общение между устройствами. Выпущен Google, апрель 2025.

### 2.8. Ограничения MAS

| Категория | Проблема | Пример |
|-----------|----------|--------|
| Ресурсоёмкость | Каждый агент = вызов LLM. 3 агента × 5 итераций = 15+ вызовов | Стоимость, задержки |
| Накопление ошибок | Ошибка одного агента усиливается по цепочке | A нашёл неверный рейс → B его забронировал |
| Отладка | Трудно понять, какой агент ошибся в длинной цепочке | Нужна трассировка каждого шага |

### 2.9. Гайд по выбору архитектуры

```
Нужен ли агент?
  → Нет → Простой API
  → Да → Сколько инструментов?
           → 2–5 → Single-Agent
           → 6+  → Нужна ли верификация?
                    → Да  → MAS + Critic
                    → Нет → Какая структура задачи?
                              → Чёткая иерархия       → Иерархическая
                              → Равноправные эксперты  → Децентрализованная
                              → Много подсистем        → Централизованная
                              → Асинхронная обработка  → Общий пул
```

**Правило:** архитектура выбирается под задачу, а не наоборот!

### 2.10. Антипаттерны — чего НЕ делать

| Антипаттерн | Проблема | Решение |
|-------------|----------|---------|
| MAS для простой задачи | Overhead > пользы | Начать с single-agent |
| 10+ агентов сразу | Невозможно отлаживать | Инкрементально: 2–3 → валидация |
| Нет критика в критичных доменах | Ошибки доходят до пользователя | Всегда добавлять Critic |
| Все агенты на дорогих моделях | Дорого и медленно | Маленькие модели для простых агентов |
| Нет трассировки | Невозможно отлаживать | Логировать каждый шаг TAO |

## 4. Практика — построение MAS для авиакомпании

Семинарная часть построена на домене τ-bench (авиалинии). Модель: `gpt-4.1-nano`. Стек: LangChain + LangGraph + Pydantic.

### 4.1. Окружение: база данных и инструменты

5 инструментов из τ-bench: `search_flights` (поиск рейсов), `get_flight_details` (детали рейса), `get_booking` (информация о бронировании), `get_policy` (политики авиакомпании), `update_booking` (изменение бронирования).

### 4.2. Часть 1: Single Agent с TAO Loop (ReAct)

Простейшая архитектура: один LLM + инструменты в цикле `Thought → Action → Observation → ... → Final Answer`.

Реализация на LangGraph с `parallel_tool_calls=False` — форсирует один вызов инструмента за шаг, делая TAO loop явным.

Системный промпт задаёт агенту роль customer support и инструкции по TAO-циклу.

**Тестирование (3 теста возрастающей сложности):**

- **Тест 1** (простой) — поиск рейсов, один вызов инструмента → агент справляется.
- **Тест 2** (мульти-шаг с явными инструкциями) — агенту прямо сказано что делать → справляется.
- **Тест 3** (естественный мульти-доменный запрос) — «Стоит ли мне отменить или перебронировать? Сравните стоимость» → агент частично проваливается, может пропустить домены.

**Вывод:** один агент с множеством обязанностей = неполное покрытие на сложных запросах. Это мотивирует переход к MAS.

### 4.3. Часть 2: Иерархическая MAS

**Что строим:** Координатор (оркестратор) + 3 специализированных агента (flight, booking, policy).

**Pydantic-схемы для структурированной коммуникации:**

```python
class SubTask(BaseModel):
    agent_name: Literal['flight_agent', 'policy_agent', 'booking_agent']
    description: str
    priority: int  # 1=highest, 3=lowest

class CoordinatorPlan(BaseModel):
    reasoning: str
    subtasks: List[SubTask]

class AgentResult(BaseModel):
    agent_name: str
    status: Literal['success', 'error', 'partial']
    result: str
    tools_used: List[str]
```

Координатор работает в 3 шага: планирование (decomposition через `with_structured_output`) → делегирование подзадач специалистам → синтез финального ответа.

**Результат:** MAS справляется с тем же мульти-доменным запросом, на котором single-agent проваливался. Компромисс: больше LLM-вызовов и медленнее, но полнее покрывает все аспекты.

### 4.4. Часть 3: Critic Agent и контроль качества

Принцип верификации. Critic Agent анализирует результаты других агентов, находит ошибки, пропущенные данные, нарушения политик. Не действует — только оценивает.

**Паттерн Reflexion** (Shinn et al., 2023):

```
Agent → Result → Critic → Approved? → Yes → User
                              ↓ No
                    Agent (retry с обратной связью)
```

**Схема обратной связи критика:**

```python
class CriticFeedback(BaseModel):
    approved: bool
    score: float        # 0–10
    issues: List[str]
    suggestions: List[str]
    reasoning: str
```

`EnhancedCoordinator` наследуется от `CoordinatorAgent` и добавляет шаг ревизии: если `score < 7`, ответ пересоставляется с учётом feedback от критика.

**Тестовый кейс:** перебронирование с ограничением по классу. Последний рейс на дату — бизнес-класс, но бронирование — эконом, а политика требует «same class only». Критик должен это поймать.

**Важно:** nano-модель не калибрована для задач оценки — может находить несуществующие проблемы. Критик должен быть мощнее оцениваемого агента (если агенты на nano, критик — на mini или выше).

**Правило:** в критичных доменах (финансы, медицина, авиация) — всегда добавлять Critic.

### 4.5. Часть 4: Бенчмарк и сравнение архитектур

Сравнение трёх подходов на одних и тех же сценариях с **fact-based объективным скорингом** — проверяется наличие конкретных ожидаемых фактов (цены, сборы, номера рейсов, политики) в ответе. Без LLM-as-judge.

| Архитектура | Описание |
|---|---|
| Single Agent (ReAct) | Один агент со всеми инструментами |
| Hierarchical MAS | Координатор + 3 специалиста |
| MAS + Critic | MAS с контролем качества |

**Ключевые результаты:**

- **Простые запросы** — Single Agent может быть быстрее и сохранять полный контекст; синтез MAS может терять мелкие детали.
- **Мульти-доменные запросы** — MAS явно выигрывает благодаря явной декомпозиции задачи (гарантированное покрытие всех доменов).
- **MAS + Critic** — добавляет задержку, но служит сеткой безопасности для качества; ловит ошибки, которые синтез MAS может пропустить.

**Когда что использовать:**

| Сценарий | Архитектура | Почему |
|---|---|---|
| Простые задачи (1–2 инструмента) | Single Agent | Быстро, дёшево, достаточно |
| Мульти-доменные (3+ инструмента) | Hierarchical MAS | Декомпозиция, специализация |
| Критичные домены | MAS + Critic | Верификация, safety net |

## 5. Итоги

### Что было построено на практике

1. **Single ReAct Agent** — TAO loop в действии (Thought → Action → Observation)
2. **Hierarchical MAS** — координатор + 3 специалиста + structured output
3. **Critic Agent** — принцип верификации, reflexion, scoring качества
4. **Бенчмарк архитектур** — измерение и сравнение подходов

### Ключевые выводы лекции

- **TAO loop** — фундаментальный паттерн, лежащий в основе любого агента.
- **ReAct > CoT** для задач с инструментами.
- **MAS** решает задачи специализации, верификации и масштабируемости.
- **4 архитектуры** MAS — нет «лучшей», есть подходящая под задачу.
- **Мультимодальность** — следующий рубеж развития агентов.

### Формула выбора

```
Нужен агент? → Сколько инструментов? → Нужна верификация?
  Нет → Pipeline    2–5 → Single Agent    Да  → MAS + Critic
                    6+  → MAS             Нет → MAS
```

**Архитектура выбирается под задачу, а не наоборот.**
